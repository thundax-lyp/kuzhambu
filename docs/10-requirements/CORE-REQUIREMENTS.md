# Core Requirements

## Purpose

Core 模块定义后台框架业务需求，负责用户主体、角色、菜单、权限资源和当前用户基础能力。

Core 不负责登录、token、认证会话和密码校验；这些能力归属 Auth。Core 不负责文件对象和业务数据审计；这些能力分别归属 Storage 和 Audit。

## Scope

覆盖：
- 后台用户主体管理。
- 管理员派发账号所需的用户资料维护。
- 用户启用、禁用和删除。
- 当前用户资料查看和修改。
- 当前用户密码修改入口编排。
- 当前用户可见菜单查询。
- 当前用户权限编码查询。
- 角色管理。
- 用户角色关系维护。
- 角色菜单关系维护。
- 菜单树管理。
- 菜单可见性管理。
- 权限编码管理。

不覆盖：
- 用户自助注册。
- 登录、登出和 token 生命周期。
- 登录标识、认证凭据和凭据锁定策略。
- 前台会员体系。
- 组织部门、岗位和多租户模型。
- 字典管理。
- 运行日志、访问日志和业务审计日志。

## Functional Requirements

- 必须支持管理员创建后台用户。
- 必须支持管理员修改后台用户资料。
- 必须支持管理员启用和禁用后台用户。
- 必须支持管理员删除后台用户。
- 必须支持管理员为用户分配角色。
- 必须支持用户查看当前用户资料。
- 必须支持用户修改当前用户资料。
- 必须支持用户发起当前用户密码修改。
- 必须支持读取当前用户可见菜单树。
- 必须支持读取当前用户权限编码集合。
- 必须支持创建、修改、启用、禁用、删除角色。
- 必须支持为角色分配菜单和权限资源。
- 必须支持维护角色拥有的用户。
- 必须支持创建、修改、删除、移动和排序菜单。
- 必须支持菜单显示和隐藏。
- 必须支持隐藏菜单承载按钮、操作列和接口级细粒度权限。
- 必须支持通过权限编码判断业务操作是否允许。

## Business Rules

- 用户主体不等于登录标识。
- 用户主体不保存密码明文、token、最近登录 IP、登录次数或登录行为明细。
- 创建用户时必须同步建立可登录所需的认证资料，认证资料的具体维护归属 Auth。
- 禁用用户后，该用户不得继续登录或访问受控功能。
- 删除用户前必须二次确认。
- 删除角色前必须清理角色与用户、角色与菜单的关系。
- 删除菜单前必须确认不会破坏角色授权和当前用户菜单展示。
- 可见菜单用于后台导航展示。
- 隐藏菜单不进入导航树，但可承载细粒度权限编码。
- 当前用户可见菜单必须按角色授权、用户状态、菜单状态和菜单可见性过滤。
- 权限编码必须稳定，不得随展示文案变化。
- admin 角色拥有全部后台管理权限。

## Acceptance Criteria

- 管理员能创建用户并派发账号。
- 禁用用户后，该用户不能登录。
- 管理员能为用户分配角色，并影响该用户的可见菜单和权限编码。
- 管理员能为角色授权菜单和权限资源。
- 当前用户只能看到自己被授权且可见的菜单。
- 当前用户权限编码能用于内容查看、编辑、删除、导出、分享、搜索和问答等业务操作判断。
- 删除角色后，相关用户不再获得该角色权限。
- 隐藏菜单不出现在导航树，但其权限编码仍可用于权限判断。
