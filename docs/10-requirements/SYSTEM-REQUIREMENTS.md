# System Requirements

## Purpose

System 域定义平台系统基础能力需求，负责后台用户、角色、菜单、权限资源、认证、token、认证会话、认证事件和业务数据变更审计。

System 不负责业务内容、文件对象、AI 编排、搜索问答或运营看板。

## Scope

覆盖：

- 后台用户主体管理。
- 当前用户资料、菜单和权限编码查询。
- 角色、菜单和权限资源管理。
- 后台账号密码登录和 portal 受控入口登录复用。
- 登录前会话、验证码、登录标识和密码凭据。
- access token、refresh token 和认证会话生命周期。
- 当前认证上下文。
- 登录、登出、刷新、认证失败和会话失效事件。
- 业务对象变更审计。
- 创建、更新、删除、启用、禁用、绑定、解绑和关系变更审计。
- 业务对象审计历史查询和筛选。

不覆盖：

- 用户自助注册。
- OAuth2 授权服务器或第三方登录。
- 前台会员体系、组织部门、岗位和多租户模型。
- 运行日志、访问日志和技术日志。
- 文件对象存储。
- 业务数据回滚和完整历史版本恢复。

## Functional Requirements

### 用户、角色和权限

- 必须支持管理员创建、修改、启用、禁用和删除后台用户。
- 必须支持管理员为用户分配角色。
- 必须支持用户查看和修改当前用户资料。
- 必须支持用户发起当前用户密码修改。
- 必须支持读取当前用户可见菜单树。
- 必须支持读取当前用户权限编码集合。
- 必须支持创建、修改、启用、禁用和删除角色。
- 必须支持为角色分配菜单和权限资源。
- 必须支持维护角色拥有的用户。
- 必须支持创建、修改、删除、移动和排序菜单。
- 必须支持菜单显示和隐藏。
- 必须支持隐藏菜单承载按钮、操作列和接口级细粒度权限。
- 必须支持通过权限编码判断业务操作是否允许。

### 认证

- 登录前必须创建 `PreAuthSession`，并生成登录 token 和验证码。
- 需要加密传输密码时，`PreAuthSession` 必须提供密码传输密钥。
- 创建、刷新或使用 `PreAuthSession` 时必须校验容量、有效性和过期状态。
- 后台账号密码登录必须校验验证码。
- 账号密码登录必须先解析登录标识，再校验认证凭据。
- 登录标识必须能定位 System 中的用户主体。
- 密码验证必须使用密码哈希校验，不得明文比较。
- 密码验证成功后必须清零失败次数、清除锁定状态并记录最近验证时间。
- 密码验证失败后必须递增失败次数，达到限制后必须锁定对应凭据。
- 登录成功后必须创建 access token、refresh token 和 `PrincipalAuthSession`。
- access token 和 refresh token 只允许明文返回一次，服务端不得持久化 token 明文。
- 有效访问请求必须建立当前认证上下文，并 touch 对应认证会话。
- refresh token 成功刷新后必须生成新的 access token，并轮换 refresh token 或标记原 refresh token 已使用。
- 登出、token 安全失效、用户禁用或按用户失效会话时，必须删除对应认证会话。
- 登录成功、登录失败、登出、token 刷新、token 撤销和会话失效必须记录认证事件。

### 业务审计

- 创建、更新、删除、启用、禁用业务对象成功后必须记录审计。
- 业务对象关系变更成功后必须记录关系变更审计。
- 审计日志必须记录业务对象类型、业务对象标识、操作动作、操作者、操作来源、请求标识、链路标识、来源地址和可读摘要。
- 审计日志必须记录变更前快照、变更后快照和字段变更摘要。
- 后台必须支持按业务对象查询审计历史。
- 后台必须支持按对象类型、对象标识、动作、操作者、来源和时间范围筛选审计日志。
- 后台必须支持读取业务对象当前审计版本、最后操作、最后操作者和最后操作时间。

## Business Rules

- 用户主体不等于登录标识。
- 登录标识不等于认证凭据。
- token 不等于认证会话。
- System 创建用户主体；认证资料由 System 域内认证子能力维护。
- 用户主体不保存密码明文、token、最近登录 IP、登录次数或登录行为明细。
- 用户主体禁用时，该用户全部登录方式不可用，已有会话也不能继续访问受控资源。
- 登录标识禁用只影响对应登录方式。
- 密码凭据锁定发生在凭据维度。
- 密码、验证码、token、secret、privateKey、传输密钥和验证码不得写入日志或审计快照。
- 认证错误响应不得泄露用户是否存在、凭据哈希、token 哈希或内部会话 ID。
- Controller 只能读取当前认证上下文，不得直接解析 token 运行态。
- 权限判断必须基于当前认证上下文和 System 提供的权限编码。
- admin 和 portal 是访问入口边界，不是两套认证业务规则。
- 可见菜单用于后台导航展示；隐藏菜单不进入导航树，但可承载细粒度权限编码。
- 权限编码必须稳定，不得随展示文案变化。
- admin 角色拥有全部后台管理权限。
- 删除用户、角色和菜单前必须进行必要确认或关系清理。
- 审计日志只记录成功提交后的业务数据变更事实。
- 一个审计日志只能对应一个业务对象。
- 默认无字段变化时不写审计日志；明确要求记录无变化动作时可以写入无字段变化审计。
- 审计写入必须与业务写操作同事务提交，审计写入失败时业务写操作必须回滚。
- 审计日志只能追加，不得被业务流程修改。
- 审计快照只包含审计展示需要的稳定字段，不得直接序列化完整业务对象。
- 手机号、邮箱等敏感展示值必须脱敏。
- 批量业务入口必须拆解为多个单对象审计事实。
- portal 不提供通用审计查询入口。

## Acceptance Criteria

- 管理员能创建用户并派发账号。
- 禁用用户后，该用户不能登录或继续访问受控资源。
- 管理员能为用户分配角色，并影响该用户的可见菜单和权限编码。
- 当前用户只能看到自己被授权且可见的菜单。
- 隐藏菜单不出现在导航树，但其权限编码仍可用于权限判断。
- 使用有效账号、密码和验证码可以登录并获得 access token 和 refresh token。
- 验证码错误、登录前会话过期或登录 token 无效时不能登录。
- 使用不存在账号或错误密码时，响应不泄露账号存在性。
- 密码连续失败达到限制后，对应密码凭据被锁定。
- access token 过期、撤销或对应会话失效后不能访问受控资源。
- refresh token 成功刷新后，旧 refresh token 不能再次使用。
- 登出后，当前 token 不能继续访问受控资源。
- 登录、登出、刷新、失败和会话失效事件可追溯。
- 创建、更新、删除、启用、禁用、绑定和解绑业务对象后，可查询到对应审计日志。
- 审计日志能追溯操作者、时间、来源、对象、动作和变更摘要。
- 敏感字段不会出现在审计快照中。
- 审计写入失败时，业务写入不会提交。
- portal 用户不能直接查询通用审计日志。
