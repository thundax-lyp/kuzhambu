# Auth Requirements

## Purpose

Auth 模块定义 kuzhambu 的认证业务需求，负责登录前会话、验证码、登录标识、认证凭据、访问 token、refresh token、认证会话、当前认证上下文和认证事件。

Auth 采用 sandwich 已验证的认证能力边界：用户主体不等于登录标识，登录标识不等于认证凭据，token 不等于认证会话。Core 负责用户、角色、菜单和权限资源；Auth 负责认证流程和认证运行态。

## Scope

覆盖：
- 后台账号密码登录。
- portal 入口需要登录时的同一套账号认证能力。
- 登录前临时会话。
- 验证码。
- 登录标识解析。
- 密码凭据校验、失败次数、锁定、过期和强制改密。
- 访问 token 创建、校验、续期和失效。
- refresh token 创建、校验、轮换和失效。
- 认证会话创建、touch、登出、失效和过期。
- 当前认证上下文建立和读取。
- 登录、登出、刷新、认证失败和会话失效事件记录。
- 用户创建、修改登录名、重置密码时的认证资料维护。

不覆盖：
- 用户自助注册。
- OAuth2 授权服务器。
- 第三方 OAuth 登录。
- 企业微信登录。
- GitHub 登录。
- 短信登录。
- MFA 二次认证。
- 邮件通知。
- 用户主体、角色、菜单和权限资源管理。
- 业务数据审计。

## Auth Concepts

- `PreAuthSession`：登录前临时会话，用于承载登录 token、验证码、密码传输密钥和登录前短期校验数据。
- `PrincipalIdentity`：登录标识，用于表达账号、手机号、邮箱等可登录标识。
- `PrincipalCredential`：认证凭据，用于表达密码哈希、失败次数、锁定状态、过期状态和强制改密状态。
- `PrincipalAuthSession`：登录后的认证会话运行态，用于表达主体、客户端、签发时间、最近访问时间、过期时间和权限集合。
- `PrincipalAccessToken`：访问 token 运行态，用于 API 请求认证。
- `PrincipalRefreshToken`：刷新 token 运行态，用于换取新的访问 token。
- `PrincipalLoginEvent`：认证事件，用于追溯登录、登出、刷新、失败和会话失效。

## Functional Requirements

### 登录前会话

- 登录前必须创建 `PreAuthSession`。
- `PreAuthSession` 必须生成登录 token。
- `PreAuthSession` 必须生成验证码。
- 需要加密传输密码时，`PreAuthSession` 必须提供密码传输密钥。
- 创建 `PreAuthSession` 时必须校验登录前会话容量。
- 刷新或使用 `PreAuthSession` 时必须校验其有效性。
- 登录成功后必须释放对应 `PreAuthSession`。

### 验证码

- 后台账号密码登录必须校验验证码。
- 验证码不存在、过期或不匹配时必须拒绝登录。
- `PreAuthSession` 无效或过期时必须拒绝登录。
- 验证码错误响应不得泄露内部会话信息。

### 登录标识

- 账号密码登录必须先解析登录标识，再校验认证凭据。
- 登录标识不存在时必须按账号或密码错误处理。
- 登录标识禁用时必须拒绝登录。
- 登录标识必须能定位 Core 中的用户主体。
- Core 用户主体不存在或已禁用时必须拒绝登录。
- 修改用户登录名时必须同步维护账号类型登录标识。

### 密码凭据

- 解析登录标识后必须读取对应密码凭据。
- 密码凭据不存在时必须按账号或密码错误处理。
- 密码凭据禁用、锁定或过期时必须拒绝登录。
- 密码验证必须使用密码哈希校验，不得明文比较。
- 密码验证成功后必须清零失败次数。
- 密码验证成功后必须清除锁定状态。
- 密码验证成功后必须记录最近验证时间。
- 密码验证失败后必须递增失败次数。
- 密码验证失败达到限制后必须锁定该凭据。
- 重置密码后必须清零失败次数和锁定状态。
- 重置密码后必须支持强制下次登录后修改密码。

### 访问 token

- 登录成功后必须创建访问 token。
- 访问 token 必须能定位认证主体和认证会话。
- 访问 token 只允许明文返回一次。
- 服务端不得持久化访问 token 明文。
- 访问 token 校验失败时必须拒绝访问。
- 访问 token 过期、撤销或会话失效时必须拒绝访问。
- 有效访问请求必须 touch 对应认证会话。

### refresh token

- 登录成功后必须创建 refresh token。
- refresh token 必须能定位认证主体、认证会话和关联访问 token。
- 服务端不得持久化 refresh token 明文。
- refresh token 过期、撤销或已使用时必须拒绝刷新。
- refresh token 刷新成功后必须生成新的访问 token。
- refresh token 刷新成功后必须轮换 refresh token 或标记原 refresh token 已使用。
- refresh token 不得重复使用。

### 认证会话

- 登录成功后必须创建 `PrincipalAuthSession`。
- `PrincipalAuthSession` 必须绑定认证主体、客户端、签发时间、最近访问时间和过期时间。
- 登录成功后必须把权限集合写入认证会话。
- 有效请求必须刷新认证会话最近访问时间。
- 登出、token 安全失效、用户禁用或按用户失效会话时，必须删除对应认证会话。
- 认证会话过期后，对应 token 不得继续访问受控资源。

### 当前认证上下文

- API 请求认证成功后必须建立当前认证上下文。
- 当前认证上下文必须能读取主体标识、入口类型、权限集合和请求标识。
- Controller 只能读取当前认证上下文，不得直接解析 token 运行态。
- 权限判断必须基于当前认证上下文和 Core 提供的权限编码。

### 认证事件

- 登录成功必须记录认证事件。
- 登录失败必须记录认证事件。
- 登出必须记录认证事件。
- token 刷新必须记录认证事件。
- token 撤销、会话失效和用户会话批量失效必须记录认证事件。
- 认证事件必须记录主体、入口、动作、时间、结果和来源摘要。

### 认证资料维护

- Core 创建用户时，Auth 必须能创建默认账号登录标识和密码凭据。
- Core 修改用户登录名时，Auth 必须能更新账号登录标识。
- Core 重置用户密码时，Auth 必须能更新密码凭据。
- Core 禁用用户时，Auth 必须能使该用户已有认证会话失效。

## Business Rules

- 平台不得提供用户自助注册入口。
- Auth 不创建用户主体；用户主体归属 Core。
- Auth 不管理角色、菜单或权限资源；这些资料归属 Core。
- 用户主体禁用时，该用户全部登录方式不可用。
- 登录标识禁用只影响对应登录方式，不等同于禁用用户主体。
- 密码凭据锁定发生在凭据维度，不发生在用户主体维度。
- 密码、验证码、token、secret 和传输密钥不得写入日志。
- 认证错误响应不得泄露用户是否存在、凭据哈希、token 哈希或内部会话 ID。
- 权限集合随认证会话存活；登出或会话失效后权限集合必须释放。
- admin 和 portal 是访问入口边界，不是两套认证业务规则。
- portal 入口不等同于无认证；需要登录的 portal 资源必须复用本模块认证能力。

## Acceptance Criteria

- 使用有效账号、密码和验证码可以登录并获得访问 token 和 refresh token。
- 验证码错误、登录前会话过期或登录 token 无效时不能登录。
- 使用不存在账号或错误密码时，响应不泄露账号存在性。
- 密码连续失败达到限制后，对应密码凭据被锁定。
- 被禁用用户不能登录，已有会话也不能继续访问受控资源。
- 被禁用登录标识不能用于登录。
- 访问 token 过期、撤销或对应会话失效后不能访问受控资源。
- refresh token 成功刷新后，旧 refresh token 不能再次使用。
- 登出后，当前 token 不能继续访问受控资源。
- 管理员按用户失效会话后，该用户已有 token 不能继续访问受控资源。
- 登录、登出、刷新、失败和会话失效事件可追溯。
